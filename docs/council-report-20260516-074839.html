LLM Council — Lideres como evaluadores

Fecha: 2026-05-16

Veredicto

El sistema esta parcialmente preparado en documentacion, pero no completamente preparado como modelo de autorizacion. Debe permitir que lideres actuen como evaluadores cuando esten explicitamente asignados a un modulo, incluso en su propio RA/SO o en otro RA/SO.

La regla tecnica debe ser: solo evalua quien esta asignado al modulo. La politica administrativa debe decidir si esa asignacion requiere revision por pares, aprobacion adicional o simple registro.

Acuerdo del consejo

Permitir el caso real
Bloquearlo empujaria el trabajo fuera del sistema.
Permisos contextuales
El rol global no basta para RA/SO, modulo y periodo.
Software + politica
El software controla acceso; la administracion define aceptabilidad.
Auditoria
Registrar quien evaluo, cuando y bajo que asignacion.

Posturas

Contrarian

El rol global es el punto debil. Sin permisos por contexto, se sobreconcede acceso o se bloquea trabajo legitimo.

First Principles Thinker

La unidad de control no es la persona, sino la accion sobre un objeto especifico.

Expansionist

Un modelo contextual habilita coevaluadores, sustitutos, revision por pares y calibracion cruzada.

Outsider

El sistema debe hacer visible que una persona puede llevar varios sombreros, o la auditoria sera confusa.

Executor

En S2, aplicar verify_module_ownership a lideres y docentes por igual en rutas de evaluacion, sin bypass por rol leader.

Primer paso

Actualizar S2 para que verify_module_ownership aplique a cualquier usuario que escriba evaluaciones de modulo y agregar tests para lider asignado a modulo propio, lider asignado a modulo de otro RA y lider no asignado.
