Infraestructura externa para RA Assessment App

Pregunta: ¿Debe el proyecto empezar ahora tareas de infraestructura externa como PostgreSQL real/staging, Caddy/HTTP, hardening, fail2ban y backups para mejorar la calidad de pruebas, o seguir solo con features de aplicación?

Contexto clave: S1-S4 implementados, 174/174 tests locales pasan, pero 5 tests PostgreSQL siguen skipped porque falta TEST_PG_URL. S5 traerá carga masiva CSV/Excel y file adapter, donde PostgreSQL, backups y despliegue real importan más.

Chairman's Verdict

Where the Council Agrees

El consejo fue unánime: iniciar infraestructura ahora, pero solo como habilitador estrecho de pruebas. El objetivo no es “terminar infraestructura”; es convertir supuestos saltados en evidencia ejecutable.

Where the Council Clashes

El desacuerdo real es el alcance. La visión expansiva ve una escalera completa de calidad: PostgreSQL, staging E2E, Caddy, TLS, backups y restore. La visión ejecutora advierte que ese camino puede convertirse en un proyecto ops sin fin. La resolución ganadora es secuenciar.

Blind Spots the Council Caught

Faltaban definición de done, timebox, ownership, política de secretos, disciplina de migraciones y estrategia de datos/reset. TEST_PG_URL debe apuntar a una base descartable o de pruebas, no a datos valiosos.

The Recommendation

Empieza infraestructura externa ahora. No sigas acumulando features solo con confianza SQLite. Pero implementa una primera rebanada mínima: PostgreSQL staging, migraciones desde DB vacía, tests PG corriendo, un E2E con PostgreSQL y prueba de backup/restore antes de cerrar S5.

The One Thing to Do First

Provisionar una base PostgreSQL 16 staging/test descartable y cablear TEST_PG_URL para que los cinco tests PostgreSQL actualmente skipped se ejecuten contra ella.

Advisor Responses

The Contrarian

Sí, iniciar ahora, pero con staging mínimo. El riesgo mayor es seguir con features sobre confianza SQLite y descubrir tarde problemas de transacciones, constraints, tipos, migrations, pooling y encoding.

The First Principles Thinker

El problema real es la calidad de evidencia. La app debe probar el universo donde vivirá: PostgreSQL 16 detrás de Caddy con backups, no solo SQLite local.

The Expansionist

La infraestructura puede ser un acelerador de producto: crea una escalera entendible de confianza, especialmente antes de S5, donde importaciones y backups elevan el riesgo.

The Outsider

Desde fuera, la señal obvia son los 5 tests skipped. Iniciar infraestructura no debe significar pausar producto, sino crear el sustrato de pruebas faltante.

The Executor

Ruta inmediata: PostgreSQL staging, TEST_PG_URL, correr tests PG, agregar un E2E con PostgreSQL, luego Caddy smoke tests y backups antes del gate S5.

Peer Review Highlights
  • La respuesta más fuerte fue la del Executor por convertir el veredicto en orden de ejecución.
  • La mayor debilidad fue la expansión excesiva de alcance: infraestructura puede volverse infinita si no se timeboxea.
  • Faltaba definir ownership, costos, secretos, reset de datos y qué tests permanecen SQLite vs cuáles deben ser PostgreSQL.
  • La primera base de pruebas debe ser descartable/test-owned para evitar destruir datos valiosos.
